# Cloudflare Release Checklist (Pages + Workers + D1)

このチェックリストは、Cloudflare 上で `Pages + Workers + D1` 構成を本番リリースする際の実務向け項目です。

## 1. Architecture

- [ ] `Pages` はフロント配信、`Workers` は API、`D1` は永続データとして責務を固定した
- [ ] 状態同期や排他制御が必要な要件について `Durable Objects` の要否を判断した
- [ ] 外部サービス連携（Auth, Payments, Analytics など）の失敗時挙動を定義した

## 2. Environments

- [ ] `dev` / `stg` / `prod` の環境を分離した
- [ ] 環境ごとに `wrangler` 設定（name, vars, bindings）を分離した
- [ ] `D1` データベースを環境別に分離した
- [ ] Preview URL と Stage URL の使い分けルールを決めた

## 3. Secrets and Bindings

- [ ] API キー、JWT シークレット、Webhook シークレットをシークレット管理に登録した
- [ ] バインディング名（例: `DB`, `KV`, `R2`）を全環境で統一した
- [ ] 本番に不要な開発用シークレットが混在していないことを確認した

## 4. D1 Operations

- [ ] マイグレーションを手順化し、実行順序を固定した
- [ ] マイグレーションの事前検証（stg で実行）を完了した
- [ ] バックアップ取得とリストア手順を文書化した
- [ ] スキーマ変更時のロールバック方針を決めた

## 5. Domain and TLS

- [ ] 本番ドメインとサブドメインの割当を確定した
- [ ] HTTPS 強制を有効化した
- [ ] `www` 正規化とリダイレクト方針を定義した
- [ ] 既存サービスと DNS / ルートが競合しないことを確認した

## 6. Caching Strategy

- [ ] 静的アセットに長期キャッシュ + ハッシュ付きファイル名を適用した
- [ ] API レスポンスのキャッシュ方針（no-store / 短 TTL）を定義した
- [ ] Cloudflare のキャッシュルールで API と静的配信を分離した
- [ ] 重要な更新が反映される無効化手順（purge）を確認した

## 7. Security

- [ ] WAF マネージドルールを有効化した
- [ ] 公開 API とログイン系エンドポイントに Rate Limiting を設定した
- [ ] 管理画面や内部 API を Access 制御で保護した
- [ ] CORS / CSP / セキュリティヘッダを確認した

## 8. Observability and Alerts

- [ ] Workers のエラーログとメトリクスを可視化した
- [ ] 監視対象として `5xx`, エラー率, p95 レイテンシを定義した
- [ ] アラート通知先（Slack / Email）を設定した
- [ ] `/health` など外形監視用エンドポイントを用意した

## 9. Release Process

- [ ] PR ごとに Preview を確認する運用を定義した
- [ ] 本番デプロイのトリガー（main merge / tag）を固定した
- [ ] リリース判定に必要なチェック項目（機能 / 性能 / セキュリティ）を定義した
- [ ] 切り戻し手順（直前バージョン再デプロイ）を検証した

## 10. Cost and Limits

- [ ] Workers 実行時間・リクエスト数の見積りを作成した
- [ ] D1 クエリ量・R2 リクエスト量の見積りを作成した
- [ ] 使用量アラートや課金上限アラートを設定した
- [ ] 想定ピーク時のコスト試算を共有した

## 11. Existing Cloudflare Asset Conflicts

- [ ] 既存サービスの Firewall / Redirect / Transform ルールと競合しないことを確認した
- [ ] 同一ゾーン内のパス競合（例: `/api/*`）を確認した
- [ ] 既存の Bot / WAF ルールが新サービスを誤検知しないことを確認した

## 12. Go-Live Day Checklist

- [ ] リリース担当、監視担当、意思決定者を明確化した
- [ ] リリース時間帯を決め、関係者へ周知した
- [ ] ロールバック条件を数値で定義した（例: 5xx 率が閾値超過）
- [ ] リリース後 30-60 分の集中監視を実施した

## 13. Post-Release (within 24h)

- [ ] エラー、レイテンシ、キャッシュヒット率をレビューした
- [ ] 問い合わせ / 不具合 / ログを確認し、優先度を付けた
- [ ] 実績コストと想定コストの乖離を確認した
- [ ] チェックリスト自体の改善点を反映した

---

## Optional: Minimal Pre-Prod Gate

本番投入前の最低基準として、次の 5 項目を必須化すると事故を減らしやすいです。

- [ ] `stg` でマイグレーション適用済み
- [ ] 重大導線の E2E または手動テスト完了
- [ ] WAF + Rate Limiting 有効
- [ ] 監視とアラートが動作確認済み
- [ ] 切り戻し手順を実行可能な状態で確認済み
